The **Normalized Difference Vegetation Index (NDVI)** is one of the most widely used remote sensing indices. It quantifies vegetation health by exploiting the fact that healthy plants reflect near-infrared light (NIR) much more strongly than visible red light.

```
NDVI = (NIR − Red) / (NIR + Red)
```

Values range from **-1** (water, deep shadow) through **0** (bare soil) to **+1** (dense, healthy vegetation). In this tutorial we compute it entirely in the browser — no backend, no cloud functions, no API keys.

---

## The Architecture

Our pipeline runs five stages, all on the client:

1. **STAC Search** — Query the [Element84 Earth Search](https://earth-search.aws.element84.com/v1) catalog to find the latest Sentinel-2 L2A scene covering the current map viewport with less than 20% cloud cover.
2. **Viewport Windowing** — Reproject the map bounds into the GeoTIFF's native UTM coordinate system and calculate the exact pixel window to fetch. This avoids downloading the entire 100 km tile.
3. **COG Fetch** — Use HTTP Range Requests to download only the bytes for the visible window from Cloud Optimized GeoTIFFs on AWS S3. The `geotiff.js` library automatically selects the optimal overview level.
4. **WebGL Compute** — Upload the Red and NIR bands as GPU textures and run a fragment shader that computes NDVI per-pixel and maps the result through a 1D colormap texture.
5. **Render** — Feed the GPU output canvas to MapLibre as an image source, with coordinates precisely back-projected from the GeoTIFF's native CRS.

## Step 1: Searching the STAC Catalog

The [SpatioTemporal Asset Catalog (STAC)](https://stacspec.org/) is a standardized way to describe geospatial data. Element84 hosts a public STAC API covering Sentinel-2 L2A imagery on AWS.

We send a `POST` request with the map's bounding box and a cloud cover filter:

```js
const response = await fetch('https://earth-search.aws.element84.com/v1/search', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    collections: ['sentinel-2-l2a'],
    bbox: [west, south, east, north],
    limit: 1,
    sortby: [{ field: 'properties.datetime', direction: 'desc' }],
    query: { 'eo:cloud_cover': { lte: 20 } },
  }),
});
const { features } = await response.json();
const scene = features[0];
```

The response includes asset URLs for every spectral band. We need two:
- **`scene.assets.red.href`** → Band 4 (Red, 665nm)
- **`scene.assets.nir.href`** → Band 8 (NIR, 842nm)

## Step 2: Viewport-Windowed COG Fetching

A naive implementation would download the entire Sentinel-2 tile (10,980 × 10,980 pixels at 10m resolution — over 200 MB per band). Instead, we calculate the exact pixel window that corresponds to the current map viewport.

The key insight is that Sentinel-2 GeoTIFFs are stored in a projected CRS (typically UTM), not in latitude/longitude. We use `proj4` to reproject the map bounds:

```js
import proj4 from 'proj4';

// Reproject map viewport bounds to the GeoTIFF's native CRS (e.g. UTM 30N)
const sw = proj4('EPSG:4326', epsg, [mapBounds.getWest(), mapBounds.getSouth()]);
const ne = proj4('EPSG:4326', epsg, [mapBounds.getEast(), mapBounds.getNorth()]);

// Convert geo-coordinates to pixel coordinates
const pxLeft  = Math.floor(((sw[0] - geoXmin) / (geoXmax - geoXmin)) * imgW);
const pxRight = Math.ceil(((ne[0] - geoXmin) / (geoXmax - geoXmin)) * imgW);
// ... (similarly for Y, remembering raster origin is top-left)
```

We then pass this pixel window to `geotiff.js`, along with a target output size. The library automatically selects the best COG overview level, fetching only the necessary bytes via HTTP Range Requests:

```js
const [redRaster, nirRaster] = await Promise.all([
  redImage.readRasters({ window: [left, top, right, bottom], width: outW, height: outH }),
  nirImage.readRasters({ window: [left, top, right, bottom], width: outW, height: outH }),
]);
```

This reduces data transfer from **~200 MB** to typically **0.5–2 MB** per viewport.

## Step 3: Computing NDVI on the GPU

This is where the WebGL magic happens. Instead of looping over pixels on the CPU, we upload the raw band data as GPU textures and run the entire NDVI calculation in a **GLSL fragment shader**.

**Crucial detail for Sentinel-2 L2A:** Since Processing Baseline 04.00 (January 2022), ESA introduced a radiometric offset of -1000 to the Digital Numbers (DN). A DN of 1000 corresponds to 0 reflectance. The shader handles this natively:

```glsl
precision highp float;
uniform sampler2D u_red;
uniform sampler2D u_nir;
uniform sampler2D u_colormap;
varying vec2 v_uv;

void main() {
  // Reconstruct original DN from normalized texture value
  float redRaw = texture2D(u_red, v_uv).r * 65535.0;
  float nirRaw = texture2D(u_nir, v_uv).r * 65535.0;

  // Skip nodata pixels
  if (redRaw < 1.0 && nirRaw < 1.0) {
    gl_FragColor = vec4(0.0);
    return;
  }

  // Subtract the radiometric offset
  float red = max(0.0, redRaw - 1000.0);
  float nir = max(0.0, nirRaw - 1000.0);

  // NDVI formula
  float denom = nir + red;
  float ndvi = denom > 0.0 ? (nir - red) / denom : 0.0;

  // Colormap lookup via 1D texture
  float lookup = clamp((ndvi + 1.0) * 0.5, 0.0, 1.0);
  gl_FragColor = texture2D(u_colormap, vec2(lookup, 0.5));
}
```

The band data is uploaded as `LUMINANCE` float textures (via the `OES_texture_float` extension), and the colormap is a pre-baked 256×1 RGBA texture. The GPU processes all pixels in parallel — the entire computation typically finishes in **under 10 ms**.

## Step 4: The Color Ramp

Our color ramp maps NDVI values to a perceptually diverging palette, baked into a 256×1 texture for GPU lookup:

```js
const STOPS = [
  { val: -1.0, r: 44,  g: 64,  b: 114 },  // deep blue  (water)
  { val: -0.2, r: 140, g: 100, b: 60  },  // brown      (bare soil)
  { val:  0.0, r: 210, g: 190, b: 140 },  // tan
  { val:  0.2, r: 180, g: 210, b: 80  },  // yellow-green
  { val:  0.4, r: 100, g: 180, b: 50  },  // light green
  { val:  0.6, r: 40,  g: 140, b: 30  },  // green
  { val:  0.8, r: 10,  g: 100, b: 20  },  // dark green
  { val:  1.0, r: 0,   g: 60,  b: 10  },  // very dark green
];
```

We linearly interpolate between adjacent stops on the CPU once to generate a 256-pixel-wide canvas, then upload it as a texture. The shader samples it with `texture2D(u_colormap, vec2(lookup, 0.5))` — the GPU's texture filtering gives us smooth color transitions for free.

## Step 5: Geographic Alignment

The GPU output canvas must be positioned precisely on the MapLibre map. Since we fetched a specific pixel window from a UTM-projected GeoTIFF, we back-project the exact pixel boundaries to WGS84 coordinates:

```js
// Back-project the pixel window corners to geo-coordinates
const actXmin = geoXmin + (winLeft  / imgW) * (geoXmax - geoXmin);
const actXmax = geoXmin + (winRight / imgW) * (geoXmax - geoXmin);
// ... convert to WGS84 via proj4 ...

map.addSource(id, {
  type: 'image',
  url: ndviCanvas.toDataURL(),
  coordinates: [topLeft, topRight, bottomRight, bottomLeft],
});
```

This produces pixel-perfect alignment between the NDVI overlay and the basemap.

## Performance Notes

With viewport windowing and GPU compute, the pipeline is dramatically faster than downloading the full tile:

| Phase | Typical Time | Notes |
|---|---|---|
| STAC search | ~400 ms | Single HTTP POST |
| COG fetch | 100–500 ms | Windowed Range Requests (vs. 3–15s for full tile) |
| WebGL compute | &lt;10 ms | Fragment shader on GPU |
| Map render | &lt;5 ms | `toDataURL` + MapLibre image source |

> **Why not a MapLibre custom layer?** We use an offscreen WebGL context rather than integrating directly into MapLibre's render loop. This avoids WebGL state management conflicts between our shader and MapLibre's internal renderer, while still achieving full GPU acceleration. The result is fed back as a standard image source — simple, reliable, and just as fast.
